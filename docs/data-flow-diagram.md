**P3A 子图数据流说明：**

- #### P3A.1 获取座位图流程（读 D7）

  - **数据流向：** `D6 座位数据 + D7 临时锁座记录 ──▶ P3A.1 ──▶ 普通用户`
  - **流程描述：** 当用户点击某一个排片场次准备选座时，系统不仅要从 `D6 座位数据`（数据库）中读取该场次的物理座位，还要同时读取 `D7` 中**已经被其他人临时锁定、但还没付钱**的座位数据。系统将这两部分数据合并后，生成一张"包含实时锁定状态"的座位画布渲染给用户，防止用户点击到正在被别人下单的座位。

- #### P3A.2 临时锁定座位流程（写 D7）

  - **数据流向：** `普通用户 ──▶ P3A.2 ──▶ D7 临时锁座记录`
  - **流程描述：** 用户在手机上选好座位并点击"确认选座"时，进入 `P3A.2`。系统会检查 `D7` 和 `D6`，确认这些座位目前都是绝对空闲的。校验通过后，系统会**立刻在 `D7` 中写入记录**（包含：场次ID、座位号、用户ID、锁定时间戳），并向用户返回"锁定成功"。

- #### P3A.3 提交订单流程（读/校验 D7）

  - **数据流向：** `普通用户 ──▶ P3A.3 ◀──▶ D7 临时锁座记录 ──▶ D10 待处理订单流水`
  - **流程描述：** 用户确认锁座成功后，点击"提交订单"进入支付排队。此时 `P3A.3` 会再次读取并校验 `D7` 中的记录，确保座位确实是被**当前这位用户**锁定的，且锁定期限没有超时。校验无误后，才会将订单送入 `D10 待处理订单流水` 中去排队。

- #### P3A.4 扣款与生成订单流程（读/删 D7）

  - **数据流向：** `D10 ──▶ P3A.4 ──▶ D1 用户数据（扣款）+ D6 座位数据（已售）+ D7 临时锁座记录（清除）+ D8 订单数据（创建）`
  - **流程描述：** 异步处理模块 `P3A.4` 从 `D10` 队列中取出订单并成功扣减用户余额（D1）后，意味着这笔交易正式做实。此时，`P3A.4` 会执行以下操作：
    1. 将 `D6 座位数据` 中对应的座位状态永久修改为"已售"。
    2. **将 `D7` 中对应的临时锁定记录彻底清除**。
    3. 在 `D8 订单数据` 中创建正式订单记录。

- #### P3A.5 查询订单状态流程

  - **数据流向：** `普通用户 ──▶ P3A.5 ──▶ D8 订单数据 ──▶ P3A.5 ──▶ 普通用户`
  - **流程描述：** 前端每 2 秒轮询 `GET /api/order/status/{orderNo}`，`P3A.5` 从 `D8` 中读取订单当前状态并返回给用户。

**资源访问：**

| 过程 | 访问的数据存储 | 访问类型 |
|------|---------------|----------|
| P3A.1 获取座位图 | D6 座位数据 | 读取 |
| P3A.1 获取座位图 | D7 临时锁座记录 | 读取 |
| P3A.2 锁定座位 | D6 座位数据 | 读取 |
| P3A.2 锁定座位 | D7 临时锁座记录 | 读取 + 写入 |
| P3A.3 提交订单 | D7 临时锁座记录 | 读取 |
| P3A.3 提交订单 | D10 待处理订单流水 | 写入 |
| P3A.4 扣款生成订单 | D10 待处理订单流水 | 读取 |

### 3.4 D7 临时锁座记录的核心作用

在系统需求分析与架构设计中，设立 D7 这个存储主要为了实现以下三个核心业务与学术目标：

#### 1. 解决高并发冲突，防止"占座不买"锁死数据库

- **核心痛点：** 电影票高并发抢票时，如果用户一选座就直接去修改关系型数据库 `D6 座位数据`，会导致数据库频繁遭遇大量的写锁和更新冲突。如果用户锁了座最后却不付钱，系统还要频繁执行回滚操作。
- **D7 的作用：** D7 作为一个**轻量级的临时缓冲区**，把"临时锁座"这种高频、不稳定的中间状态从持久化的 `D6 座位数据` 中剥离出来。只有真正付了钱的座位才会去动 D6，大大减轻了核心数据库的压力。

#### 2. 实现"倒计时自动释放"的业务逻辑

- **核心痛点：** 电影院不允许用户无限期占用座位。通常用户选座后，系统会提示"请在 15 分钟内完成支付，否则座位将被释放"。
- **D7 的作用：** D7 专门维护了`锁定时间`字段。当 `P3A.1` 或 `P3A.4` 读取 D7 时，如果发现 `当前时间 - 锁定时间 > 15分钟`，系统在逻辑上直接判定该锁定失效（在物理上通常由定时任务或缓存过期机制自动擦除 D7 记录）。这就完美支撑了"超时自动释放座位"的需求。

#### 3. 保证"一开多放"的原子性与排他性

- **核心痛点：** 当两个用户在同一万分之一秒内，同时点击了同一座位并提交，系统如何保证只有一个人能拿到？
- **D7 的作用：** 在 `P3A.2` 阶段，D7 充当了**逻辑排他锁**。谁的数据先写入 D7，这个座位在未来的 15 分钟内就归谁所有，后提交的用户在读取 D7 时会直接触发"座位冲突"并被拒绝，从数据流的源头上杜绝了"多个人买到同一张票"的超卖惨剧。
